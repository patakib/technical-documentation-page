Introduction

Django is a backend framework for web development, based on Python programming language. Django is a fully open-source and free project, which you can contribute to. The framework is a perfect choice for database-heavy web projects. Django's first release was in 2005, it is maintained by the Django Software Foundation.

Django comes with a built-in admininstrative site, which is a practical and easy feature to support CRUD (create, retrieve, update, delete) operations. It also has an ORM (Object-relational Mapper) that manages relational databases through "models", which are actually Python classes. Django has a simple web server for development purposes.

Its community is active and huge. There is a regular conference since 2008, called DjangoCon.

Even if this is a site where we introduce Django, the single source of truth is the official documentation. Please check that now if you haven't done it yet and do it continuously on your learning journey, it is compact, easy to read, very logical - absolutely fantastic.

Getting Started

First of all, as Python is the language you'll use during programming in Django, let's check whether you have Python on your local computer:
>>>python

If you have Python installed, you can create a virtual environment and activate it by typing: >>>python -m venv myenvironment
>>>source myenvironment/bin/activate

Afterwards you can install Django in your environment: >>>python -m pip install Django

Then you can create your first project: >>>django-admin startproject myproject There is a special terminology here in Django, where we have projects and in each project we can have multiple applications.

File Structure

You have created your first project, now let's take a look at the files Django created for you.
It should be like that:

  • myproject/
    • manage.py
    • myproject/
      • __init__.py
      • asgi.py
      • wsgi.py
      • settings.py
      • urls.py

manage.py is the core of the project, you can give orders to your project via CLI with this file.
You can run the development server with this file: >>>python manage.py runserver settings.py contains the main settings of your project including database, debug, installed apps and path options.

You can start your first app by typing: >>>python manage.py startapp myfirstapp It has created further files in your project:

  • myfirstapp/
    • __init__.py
    • admin.py
    • apps.py
    • models.py
    • tests.py
    • views.py
It has also created another folder called migrations.

Models

Django has an object-relational mapper which we can interact with by editing models.py. Here we can create database tables by creating Python classes. Django supports different database management systems, such as MySQL, PostgreSQL or Oracle.

Each class within models.py is a table in our database (which we can define and plug-in to our project by editing settings.py). These classes are derived from models.Model superclass.

This is an API which makes our table creation very effective and easy. Within your classes you really just need to define the attributes which will be the actual fields in the table: from django.db import models

class Student(models.Model):
  first_name = models.CharField(max_length=50)
  last_name = models.CharField(max_length=80)

What have we done exactly?
We basically created the Student table (like typing CREATE TABLE STUDENT in SQL) and added the fields first_name and last_name. We defined the field type and the max possible length of the text.
This is the way you can create tables and fields. There are tons of options for field types you can choose from in Django.

We have to talk about one important thing when discussing models in Django, this is the migration.
When you run this command in the shell: >>>python manage.py makemigrations You tell Django to create the SQL parameters based on the models and prepare everything before run the database updates. Django stores these parameters in the migrations folder.

After that you can run this command: >>>python manage.py migrate Here you tell Django to apply the migrations you defined previously. This is the moment when your database reflects your models and everything is updated with respect to the database.

Admin

Django comes with a built-in administration site. you just have to run the following command in the shell: >>>python manage.py createsuperuser Here you will have to provide the superuser's name, email and password.

If you start the development server by typing: python manage.py runserver And open the localhost/admin (which is normally http://127.0.0.1:8000/admin/), you can see the admin page. In order to see the Student model here, you have to register it.

Open up your admin.py file and add: from .models import Student

admin.site.register(Student) If you refresh the admin site, you will see the Student model.

Here you can add actual students to the database, you can update their attributes and delete them.

Views

views.py is the file where you can build the logic of what you want to display for the user. You can define functions and each of them will refer to a subpage of the website. The goal of these view functions is to provide a logic to the system behind, which information you want to retrieve from the database on that particular site.

You also can think of views as the connector between model and html templates. The database model stores your data (the structure is provided by you in models.py), the html template defines how the data is to be displayed on the front-end. Django views connects them together in a special logic which you create in those functions.

Django views take an HTTP request as a parameter, get the data from the database and render a template as reply to the request.

Here is an example: def student_list(request):
  students = Student.objects.all()
  return render(request, 'myfirstapp/list.html',{'students':students}) It gives you back the list of all students registered in the Student table, when you request list.html.

Templates

Templates are the html files which define how the data should be displayed on the client side.
You can dinamically call the data in the template, you retrieve from models.py based on the logic of views.py for this purpose there are some important parts of Django's template language:

  • tags
  • variables
  • filters

Template tags like {% for %}or {% if %} support the Python logic behind and they have a closing tag as well - {% endfor %} and {% endif %} respectively.

If you want to display values dinamically, you should store those values in variables in views.py and Django provides with the result of the variable when you call them in the template.
An example for template variables in this case could be {{ student.last_name }}, where you get the last name of the student(s).

Template filters are suitable for modifications on the variables, for instance {{ student.first_name|upper }}, where you get the first name of the student(s) converted to uppercase.

URLs

There is a urls.py file in the project folder, but generally you need to create another one in your app folder with the same name. The logic behind this is to register all the urls in the app and you can easily plug-in those urls from each app in the project.

So let's create a urls.py in your myfirstapp folder and put this code into it: from django.urls import path

from . import views

urlpatterns = [path('list.html', views.student, name='list')]

Then open your another urls.py file from your project folder and type: from django.contrib import admin
from django.urls import include, path

urlpatterns = [
  path('myfirstapp/', include('myfirstapp.urls')),
  path('admin/', admin.site.urls),
]
Congratulations!
You have just created a basic Django app :)
